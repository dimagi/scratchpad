Configurable Case Search Endpoints

A new system for defining case search endpoints with structured, visual filter logic — backed by auto-generated project database tables that enable efficient relational queries.

1

Project Database Tables

For each case type in a project, the system automatically generates a database table based on the project's data dictionary. No manual configuration is needed.

Each case property becomes a column with its declared data type (text, number, date, etc.). Case relationships — parent/child, host/extension — become foreign key links between tables, enabling direct JOINs instead of the current multi-step lookups.

Why not Elasticsearch?  ES stores all properties in a flat, untyped structure and can't do JOINs. Querying across case relationships currently requires multiple round trips with a hard limit of 500k parent cases. Project database tables eliminate both constraints.
2

Populating Tables

Tables are kept up to date via CommCare's existing change feed — the same mechanism that currently feeds Elasticsearch and UCR.

When a case is created or updated, a single row is inserted or updated in the appropriate table. Values are automatically cast to their declared types (a date string becomes a real date column, a number string becomes a numeric column). If a value can't be cast, the raw text is still stored.

Because the data is normalized (each case type in its own table), there are no related-case lookups at write time. Updating a parent case only touches the parent's row — child rows don't need updating. This makes writes significantly cheaper than the current ES or UCR approach.

3

Configuring an Endpoint

An admin configures a case search endpoint by specifying a case type, defining input parameters, and building filter logic using a visual query builder.

Input parameters are named, typed values that the endpoint expects to receive at request time. For example:

search_province text
Province to filter on
min_age number
Minimum patient age
cutoff_date date
Only cases after this date

Filter logic is composed visually using the query builder. The admin selects case properties, chooses operations (equals, greater than, starts with, etc.), and specifies values. Conditions can be grouped with AND, OR, and NOT at any nesting depth.

Each value in a filter condition comes from one of three sources:

Constant
A fixed value set by the admin
"active"
Parameter
Filled in at request time
search_province
Auto-value
Resolved automatically
today()

Auto-values include today() now() user.username user.location_ids and other context about the requesting user — available without the admin defining them.

Available operations depend on the data type. The backend declares which filter operations it supports for each type. The UI only shows valid combinations — for example, "starts with" appears for text fields but not date fields, while "before / after" appears for dates but not text.

The filter configuration is saved as a structured specification. It is backend-agnostic — the same specification can be executed against Elasticsearch or SQL.

4

At Request Time

When a mobile application or integration sends a request to the endpoint, the system loads the stored filter specification, binds the parameter values from the request, resolves auto-values (current date, user info), and hands the complete filter tree to the backend for execution.

With the project database backend, cross-relationship filters become direct JOIN queries — for example, "find all patient cases where their household's district is Kamuli" is a single efficient query instead of multiple round trips.

Backend flexibility. Because the filter specification is backend-agnostic, the same endpoint configuration can be executed against Elasticsearch (today) or the new project database tables (coming soon) — with no changes to the endpoint definition.
